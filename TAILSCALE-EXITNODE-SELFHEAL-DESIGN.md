# Tailscale exit-node self-heal — design (no-brick)

_Generated 2026-06-14 from a grounded trace of the route/exit-node lifecycle._

## Problem
Exit-node split routes (0/1+128/1) die on sleep/wake or a connectivity blip, nothing re-installs them, panic_reset destroys the intent, and the machine wedges until reboot.

## Headline

The root cause is that the SuperManager-only 0/1+128/1 split-defaults have no persisted intent and no reconcile loop, while panic_reset destroys the only intent record (tailscaled's pref) on a transient blip. The fix mirrors the proven auto_reconnect pattern: persist a 'desired exit node' in /var/lib/supermanager, make panic_reset fail open (remove routes, keep the pref), and add an always-on 30s tailscale reconciler that re-installs the split routes ONLY after re-confirming the peer is set+reachable (test_exit_reachability) on the current (post-wake) utun — otherwise stay safely on the local uplink. Self-heal lives in the LaunchDaemon's auto_reconnect tick so it covers the GUI-closed case the user actually hits.

## Principles

- NO-BRICK invariant: install 0/1+128/1 via the tailscale utun ONLY after BOTH gates pass — (a) `tailscale status --json` shows a Peer with ExitNode==true && Online==true matching the desired ID/IP, AND (b) test_exit_reachability() (tailscale.rs:655) returns a 2xx/3xx /32 probe. Never blind-install.
- FAIL OPEN, never fail closed: every teardown path (panic_reset, sweep, remove_exit_routes) must leave a working local-uplink 0/0 default. Removing 0/1 is always safe; the brick is only ever caused by ADDING 0/1 into a dead tunnel.
- Intent is sacred and must outlive the data plane: persist the desired exit node to /var/lib/supermanager (survives reboot/helper-restart/blip). A transient event may tear down ROUTES but must never destroy INTENT — so the watchdog-fired panic_reset must stop clearing the tailscaled pref.
- Reconcile, don't react blindly: a single always-on loop (the LaunchDaemon auto_reconnect 30s tick) is the only place self-heal belongs — it covers the GUI-closed case; GUI observers and tab-gated pollers do not.
- Re-detect, never assume: on wake the tailscale utun is renumbered; the reconciler must re-run detect_tailscale_utun each tick and gate on the CURRENT utun, never the pre-sleep one.
- The health probe must not depend on the tunnel it guards: when an exit node is active, measure local-uplink/control-plane health, not exit-peer egress, so a slow-but-healthy peer cannot trigger its own destruction.
- Bounded, idempotent, observable: every reconcile action is idempotent (no-op when healthy), pauses the connectivity watchdog during its disruptive window, and logs its gate decisions.

## Changes (ordered, low-risk-first)

### 1. Persist desired-exit-node intent in /var/lib/supermanager (new tailscale_state.rs)  [risk: low]
- **Files:** NEW supermanager-helper/src/tailscale_state.rs; main.rs (mod decl + call in tailscale_install_exit_routes / tailscale_remove_exit_routes RPC arms ~lines 732-746)
- **What:** Create a small persisted record mirroring auto_reconnect.rs's STATE_PATH/persist pattern: const DESIRED_PATH="/var/lib/supermanager/tailscale_desired.json" holding { desired: bool, exit_node_id: String, exit_node_ip: String, updated_unix: u64 }. Write `desired=true` + the resolved ExitNodeID/IP the moment tailscale_install_exit_routes SUCCEEDS (after routes are actually up, in the RPC arm at main.rs:733), and write `desired=false` (clear) the moment tailscale_remove_exit_routes is called from the INTENTIONAL clear path. The ID/IP is read from `tailscale debug prefs` via the existing cli_candidates+socket plumbing already in panic_reset (tailscale.rs:950-968), so no new dependency. This is the single source of truth that survives reboot/helper-restart/panic_reset, which today only tailscaled's own pref holds (and panic_reset wipes that).
- **No-brick:** Pure file I/O under /var/lib/supermanager. Writes/reads JSON; touches no routes, no tailscaled prefs, no kernel state. It cannot install or delete any route, so it cannot black-hole anything. Worst case the file is missing/corrupt → reconciler (change 3) treats it as 'no desired exit node' and does nothing, identical to today's behavior.
- **Verify:** Set an exit node via the app, confirm /var/lib/supermanager/tailscale_desired.json appears with desired=true + correct ExitNodeID; clear it, confirm desired=false. Reboot, confirm file persists. Unit-testable in isolation (serialize/deserialize round-trip).

### 2. Make panic_reset FAIL OPEN: remove routes only, keep tailscaled pref intact  [risk: medium]
- **Files:** tailscale.rs panic_reset (lines 934-1024): drop/parameterize the `tailscale set --exit-node= --accept-routes=false` block at lines 957-968; remove_exit_routes preserves EXEMPTION_STATE_FILE handling but pref is left set
- **What:** Split the current nuclear behavior. Default panic_reset path (the one the watchdog fires) should: (a) call remove_exit_routes to drop the 0/1+128/1 split-defaults so egress fails open to the local uplink, and (b) DHCP-renew — but NOT clear the tailscaled exit-node pref. Keeping the pref set means the reconciler (change 3) still knows the user wants the exit node and can re-test-and-reinstall once internet returns. Keep a separate explicit 'hard clear' (used only by the user-initiated panicResetTailscale menu item and intentional clear) that also clears the pref + the persisted desired-state. Concretely: add a `clear_pref: bool` field to PanicResetArgs (default false for the watchdog caller at connectivity_watchdog.rs:162; true for the Swift menu RPC). The desired-state file (change 1) is NOT cleared by the watchdog-fired panic_reset.
- **No-brick:** Strictly safer than today: it still REMOVES the /1 routes (the documented brick is INSTALLING 0/1 into a dead tunnel, never removing it), and removing 0/1 restores the local-uplink 0/0 default = working internet. DHCP-renew is unchanged. Not clearing the pref changes no routing state at all — the pref is daemon metadata; with the /1 routes gone, a 'set' pref forwards nothing on its own. So the machine is online on the local uplink either way; the only difference is intent is preserved for self-heal.
- **Verify:** Simulate a 6s blip (block 1.1.1.1) while on exit node; confirm after the watchdog fires: 0/1+128/1 gone, internet works via en0, AND `tailscale debug prefs` still shows ExitNodeID set (pref NOT cleared). Confirm the user menu 'Panic reset' still fully clears pref + desired-state file.

### 3. Add a tailscale reconciler to the always-on auto_reconnect 30s tick (the self-heal core)  [risk: medium]
- **Files:** NEW pub fn tailscale::reconcile_exit_node() in tailscale.rs; auto_reconnect.rs watchdog_loop (after the per-profile loop, ~line 186) calls it each tick; mirrors the full_tunnel_routes_present() pattern at auto_reconnect.rs:297
- **What:** Each 30s tick (always-on, GUI-closed-safe, LaunchDaemon), run reconcile_exit_node(): (1) read desired-state (change 1); if desired=false → no-op. (2) If desired=true, check `full-tunnel routes present on the tailscale utun` — reuse detect_tailscale_utun (tailscale.rs:820) and route_iface_family("0.0.0.0/1","-inet") to confirm 0/1 points at the CURRENT tailscale utun (re-detected, not the pre-sleep one). If present → no-op (healthy). (3) If desired but routes ABSENT, gate re-install behind BOTH: (a) `tailscale status --json` shows a Peer with ExitNode==true && Online==true matching the desired ExitNodeID/IP (re-set the pref via CLI first if tailscaled lost it), AND (b) test_exit_reachability() returns success. Only if BOTH pass → re-run install_exit_routes (which re-snapshots underlay exemptions fresh via collect_tailscaled_underlay_ips, fixing the deleted-EXEMPTION_STATE_FILE problem). If either gate fails → do NOTHING this tick (stay fail-open on local uplink), retry next tick. Pause the connectivity_watchdog (connectivity_watchdog::pause_for) for the duration of the reinstall, exactly as setExitNodeWithSafety does, so the reinstall's transient disruption can't trip panic_reset.
- **No-brick:** install_exit_routes is called ONLY after the exact two-gate check that the safe user flow uses (test_exit_reachability passing = a real /32 probe through the peer returned 2xx/3xx, AND status shows the peer ExitNode+Online). This is strictly the documented no-brick precondition: never install 0/1 unless the exit node is confirmed set AND reachable. If the peer is unreachable post-wake/post-blip, BOTH gates block install and the machine stays on the working local uplink (fail open). The reconciler never removes a working local default; install_exit_routes itself rolls back its exemptions and partner /1 on any failure (tailscale.rs:430,439).
- **Verify:** Sleep/wake on exit node with GUI closed: within 30s confirm 0/1+128/1 reinstalled on the new utun and egress goes through the peer again, with NO reboot. Then test peer-down case: power off the exit peer, sleep/wake → confirm reconciler does NOT install (gates fail), internet stays up via en0, and once the peer returns the next tick reinstalls. Check helper log for the gate decisions.

### 4. Pause the connectivity watchdog on wake (both wake paths)  [risk: low]
- **Files:** main.rs wake detector (line 214-215) and system_wake RPC (lines 952-958): add connectivity_watchdog::pause_for(~45s) before reset_snapshot/sweep; optionally AppState+SleepWake.swift handleDidWake (line ~114) belt-and-braces
- **What:** On every wake, arm a ~45s watchdog escalation pause (the API already exists: connectivity_watchdog::pause_for, connectivity_watchdog.rs:60) so the post-wake network-settle gap (interface reconfig, tailscaled re-handshake, reconciler's reinstall window) cannot accumulate 3 probe misses and fire panic_reset before the reconciler (change 3) has a chance to re-test-and-reinstall. This is the wake analog of the 30s pause setExitNodeWithSafety already arms (AppState+Tailscale.swift:386).
- **No-brick:** pause_for only SUPPRESSES escalation (force_restore / panic_reset) for a bounded window; it never installs or deletes a route. The route_guardian and DNS watchdog keep running underneath (per the existing pause doc). Strictly reduces destructive action during the fragile wake window; it cannot create a black-hole because it does nothing to the routing table.
- **Verify:** Sleep/wake and grep the helper log: confirm 'watchdog escalation paused' is logged on wake and no panic_reset fires during the settle window. Confirm escalation resumes (and still protects against a genuine sustained outage) after the window.

### 5. Stop the watchdog probe from self-defeating: exempt the probe target and loosen timeout  [risk: medium]
- **Files:** connectivity_watchdog.rs probe_internet (lines 191-198) and escalation timing (line 155); coordinate with install_exit_routes exemption set (tailscale.rs:396-416)
- **What:** Two parts. (a) When an exit node is active, the 1.1.1.1:443 probe currently routes THROUGH the utun (no /32 exemption), so exit-peer latency reads as 'internet down' and the watchdog destroys a healthy node. Either add a /32 exemption for the probe target (1.1.1.1 via local gateway, added to the exemption set in install_exit_routes so the watchdog measures LOCAL uplink health, not exit-peer egress), OR probe the tailscale control/DERP/underlay path separately when an exit node is active. (b) The 1s connect budget (nc -G 1 -w 1) is far tighter than the documented 3-6s legitimate DERP/peer latency and the 8s test_exit_reachability budget (tailscale.rs:688-707); raise the per-probe budget and/or the miss threshold so a healthy-but-slow peer doesn't trip the 6s panic_reset. This decouples 'exit-path slow' from 'local uplink down'.
- **No-brick:** Adding a /32 exemption for 1.1.1.1 via the LOCAL gateway only makes one host bypass the tunnel toward the working uplink — it cannot black-hole (it points at the live local gateway, never at the utun). Loosening probe timeout/threshold only makes the destructive panic_reset fire LESS often / later; it never installs a route. Even if the watchdog still eventually fires, change 2 made panic_reset fail-open and change 3 reconciles it back, so no wedge.
- **Verify:** On exit node, introduce 3-5s of exit-peer latency (e.g. DERP-only peer) and confirm the watchdog no longer panic_resets a healthy node; confirm a genuine local-uplink outage (pull WiFi) still escalates. Verify the 1.1.1.1 /32 exemption appears in EXEMPTION_STATE_FILE while on exit node and is cleaned up on clear.

### 6. Add a tailscale-active guard to sweep_stale_configs (defense-in-depth for the wake sweep)  [risk: low]
- **Files:** strongswan.rs sweep_stale_configs early-return (lines 50-53), mirroring has_established_strongswan_sa; delete_full_tunnel_routes (line 654)
- **What:** The wake sweep's only short-circuit is has_established_strongswan_sa() (strongswan.rs:50). Add a parallel 'tailscale exit node is desired+active' check so the sweep skips delete_full_tunnel_routes when the persisted desired-state (change 1) says an exit node should be up. This removes reliance on the timing-fragile per-route foreign_tunnel_ifaces lookup (which is empty during the post-wake window when tailscaled hasn't re-published the 100.x route, strongswan.rs:849). Belt-and-braces with change 3 — even if the reconciler hasn't ticked yet, the sweep won't strip a surviving/recovering exit node.
- **No-brick:** This only makes the sweep delete FEWER routes (it adds a skip condition). It never installs a route. The worst case is a stale 0/1 pointing at a dead utun survives one extra sweep cycle — but change 3's reconciler re-detects the current utun and the route_guardian keeps the local 0/0 default alive, so egress still works via the longest-prefix local default if the /1's utun is dead (kernel falls through). It cannot create a black-hole that change 3 won't reconcile within 30s.
- **Verify:** Sleep/wake on exit node and confirm the helper log shows the sweep skipping (or the foreign guard keeping) the tailscale /1 routes; confirm strongSwan and WG full tunnels are still correctly swept when no tailscale exit is desired.

## Self-heal design

Intent is persisted by change 1 to /var/lib/supermanager/tailscale_desired.json (persistent, unlike the volatile /var/run EXEMPTION_STATE_FILE), written on successful install_exit_routes and cleared only on intentional user clear — mirroring auto_reconnect.rs STATE_PATH (line 38) + persist (line 152). Reconciliation runs in the always-on LaunchDaemon auto_reconnect 30s tick (so it works GUI-closed, the case the user hits) via a new tailscale::reconcile_exit_node() called alongside the existing per-profile checks, directly analogous to the full_tunnel_routes_present() replay at auto_reconnect.rs:297. Each tick the reconciler reads desired-state; if desired=false it no-ops. If desired=true it re-detects the CURRENT tailscale utun (detect_tailscale_utun, tailscale.rs:820 — handles post-wake renumbering) and checks whether 0/1 already points at it (route_iface_family). If routes are present it no-ops (healthy). If routes are ABSENT (post-sleep utun teardown, or post-blip panic_reset removal) it applies the REACHABILITY GATE before any install: it confirms via `tailscale status --json` that the desired peer shows ExitNode==true && Online==true (re-setting the pref via the bundled CLI on /var/run/tailscaled.socket first if tailscaled lost it), AND runs test_exit_reachability() (tailscale.rs:655); only if BOTH pass does it call install_exit_routes (which re-snapshots underlay exemptions fresh via collect_tailscaled_underlay_ips, repairing the deleted EXEMPTION_STATE_FILE), with the connectivity watchdog paused for the reinstall window. If EITHER gate fails it does nothing and retries next tick, leaving the machine fail-open on the local uplink. On wake specifically, change 4 arms a ~45s watchdog pause and change 2 ensures any watchdog-fired panic_reset left the pref (and the desired-state file) intact, so the very next reconcile tick has everything it needs to bring the exit node back automatically — no reboot, no manual re-toggle.

## First safe step

Implement change 1 alone: add tailscale_state.rs that persists { desired, exit_node_id, exit_node_ip } to /var/lib/supermanager/tailscale_desired.json, and write it (desired=true) on successful tailscale_install_exit_routes / (desired=false) on intentional tailscale_remove_exit_routes, mirroring auto_reconnect.rs:38/152. This is pure JSON file I/O that touches no routes and no tailscaled prefs, so it physically cannot black-hole the machine; it just starts capturing the user's intent so every later self-heal step has a source of truth to reconcile against. It is independently shippable and a no-op for routing until change 3 reads it.

## Open questions

- Where exactly should the resolved ExitNodeID/IP be sourced when writing desired-state on install — from the Swift side (it already set the pref) passed through the install RPC params, or read by the helper itself via `tailscale debug prefs` at install time? Reading helper-side keeps the helper self-sufficient (no GUI dependency) and is preferred, but adds a CLI round-trip in the install path.
- Reconcile cadence: is the 30s auto_reconnect tick fast enough for the wake case, or should wake additionally trigger an immediate one-shot reconcile (e.g. system_wake RPC kicks reconcile_exit_node() after the sweep, with a short delay for tailscaled to re-handshake)? A wake-triggered reconcile would cut the worst-case re-establish latency from ~30s to a few seconds.
- For change 5, is exempting the watchdog probe target (1.1.1.1 /32 via local gateway) acceptable, given that it means the watchdog no longer detects 'exit-peer egress is down' at all — or do we want a SECONDARY probe through the peer that only DISABLES (fails open) rather than panic_resets? Need to decide what 'exit node is down but local uplink is up' should do: silently fall back vs. notify.
- Should the reconciler also handle the 'auto:any' exit node selection (ipOrAuto == "auto:any", AppState+Tailscale.swift:414) where no specific ID is pinned? The status-based ExitNode==true gate still works, but the persisted desired-state needs an 'auto' marker rather than a fixed ID/IP.
- IPv6: install_exit_routes adds ::/1+8000::/1 best-effort; should the reconciler's route-presence check and reachability gate consider v6 at all, or remain v4-gated like full_tunnel_routes_present() (strongswan.rs:689)?
- Interaction with a simultaneously-active WG/OpenVPN/IKEv2 full tunnel that ALSO owns 0/1: the reconciler must not fight the strongSwan/foreign ownership of those routes. Need to confirm the reconciler defers when a foreign tunnel currently owns 0/1 (it should treat that as 'not my routes, do nothing').

## Scenario walkthroughs (evidence)

### sleep-wake-on-exit
End state: Route table: no 0.0.0.0/1 or 128.0.0.0/1 pointing at the tailscale utun (they died with the pre-sleep utun and are never re-installed). Egress: all traffic exits the local physical uplink — the exit node is silently bypassed (fail-open, so internet itself works). Exit-node state in tailscaled: either still SET (ExitNodeID/IP non-empty) if no blip pushed the watchdog to panic_reset = a stable 'configured-but-not-enforced' wedge; or CLEARED if a post-wake probe gap fired panic_reset (tailscale.rs:961-967), which also deleted the /32 exemptions and unlinked EXEMPTION_STATE_FILE. UI shows the exit node still selected. No reconciler detects the pref/route mismatch; no path re-runs test_exit_reachability -> install_exit_routes. State is frozen until manual re-toggle or reboot.
Reboot needed because: Because the 0/1+128/1 split-defaults are a SuperManager-only artifact (open-source tailscaled does not install them) and there is NO self-heal/reconcile loop for them and NO persisted 'desired exit node' record on the helper side. On wake the old utun (and its interface-scoped /1 routes) is destroyed; the only code that re-creates those routes, install_exit_routes (tailscale.rs:358), is reachable exclusively from the user-initiated Swift setExitNodeWithSafety flow (RPC at main.rs:732, called from AppState+Tailscale.swift:455). Every wake path (main.rs:198-219, main.rs:945-962, AppState+SleepWake.swift:88-128, AppState.swift:362-378) only sweeps/refreshes/polls and never re-installs. So the routes stay gone. If a post-wake probe gap additionally trips panic_reset, the tailscaled exit-node pref is cleared too, so even tailscaled's own intent is lost. The user has no in-app affordance wired to a reconcile, so the practical way to get the utun recreated AND the SuperManager route-install flow re-run from scratch is a reboot (or manually re-toggling the exit node, which re-enters setExitNodeWithSafety -> test_exit_reachability -> install_exit_routes).

### blip-on-exit
End state: Route table: tailscale split-defaults (0.0.0.0/1, 128.0.0.0/1, ::/1, 8000::/1) DELETED; only the local en0 0/0 default remains (re-leased by DHCP). Underlay /32 exemption host routes DELETED and EXEMPTION_STATE_FILE unlinked. tailscaled exit-node state: pref CLEARED (ExitNodeID/ExitNodeIP empty) by `tailscale set --exit-node=`; accept-routes=false; Peer[].ExitNode==true on no peer. Egress: FAIL OPEN to the local uplink — internet works, but all traffic bypasses the exit node (no longer enforced). UI: may still show the exit node as 'selected' from stale state until next refresh, but it is not enforced anywhere. This is a stable wedged state that persists indefinitely.
Reboot needed because: The exit node does not come back by itself because of three compounding gaps, all verified in code. (1) NO RECONCILER / SELF-HEAL: the 0/1+128/1 split-defaults are a SuperManager-only artifact (open-source tailscaled does not install them); install_exit_routes (tailscale.rs:358) has exactly ONE caller — the user-initiated setExitNodeWithSafety (AppState+Tailscale.swift:455). No timer, observer, wake path, or helper loop ever re-runs it. The helper's auto_reconnect 30s tick reconciles WG/OVPN/IKEv2 ('SA established but full-tunnel routes missing — forcing replay', auto_reconnect.rs:297) but has NO tailscale equivalent. (2) INTENT DESTROYED: panic_reset additionally runs `tailscale set --exit-node=` (tailscale.rs:961-967), erasing the daemon pref that was the only persisted record of the desired exit node, and deletes EXEMPTION_STATE_FILE — so even a hypothetical reconciler would have nothing to read back and could not safely re-snapshot exemptions. (3) SELF-DEFEATING TRIGGER: the watchdog probe to 1.1.1.1:443 is routed THROUGH the utun while the exit node is up (no /32 exemption for the probe target, connectivity_watchdog.rs:191-198), and its 1s timeout (nc -G 1 -w 1) is far tighter than the documented 3-6s legitimate DERP/peer latency and the 8s test_exit_reachability budget (tailscale.rs:688-707) — so a transient exit-path slowdown is indistinguishable from real ISP loss and trips panic_reset on a HEALTHY exit node. Because the routes are gone, the pref is gone, the exemption snapshot is gone, and nothing reconciles, the machine sits fail-open on the local uplink until the user manually re-selects the exit node or reboots. The proper no-brick fix: persist desired-exit-node in the helper; make panic_reset remove ROUTES ONLY (leave the daemon pref intact, fail open to local uplink); add a tailscale reconciler in the auto_reconnect tick that, when desired-exit-node is set but routes are absent, re-detects the current utun (detect_tailscale_utun), confirms a Peer with ExitNode==true && Online==true matches the pref, runs test_exit_reachability, and only THEN calls install_exit_routes — never installing 0/1 blind.
