## Helhetsvurdering

Solid-but-structurally-flawed. The individual surfaces are unusually well-built for an internal tool — VPN, SSH, and Tailscale have real depth (kill-switch, live telemetry, exit-node safety nets), Fleet is a genuine cross-customer dashboard, and the Compliance/Provisioning render-diff-deploy loops are substantive. The code is also self-aware: many of the exact defects the owner suspected are documented in code comments (the UniFi "K3" reconciliation TODO, the "wrong-scope bug C1" warning, the VpnStatusDot "copy-pasted colour mappings" note). But the app has one load-bearing architectural defect that the polish can't paper over: there is no shared Customer→Site→Host spine. The same FortiGate is identified four incompatible ways — a free-text host.group string, an IP stuffed into Site.hostIds, an IP-keyed findings store, and a record-id-keyed compliance store — and nothing reconciles them. That single root cause produces the owner's headline symptom (Compliance says "No compliance-capable hosts" for a customer whose FortiGates are sitting right there in Provisioning), makes Fleet silently blind to compliance failures, structurally breaks Provisioning diff/deploy for every auto-discovered host, and means the global customer picker means four different things across tabs and nothing in three others. On top of that spine problem sits a pervasive secondary pattern: six of eight sections open onto a dead "Select X to begin" pane in the app's largest panel, every time, even though the data to fill them is already in memory. Neither problem is cosmetic; both cause real rework and real wrong-target risk (stale Provisioning render can deploy Site A's config to Site B; discovered hosts get group="Discovered" and vanish from their customer forever). This is not a struggling app — it's a strong app with a missing backbone. Fix the host-identity spine and the empty panes and it goes from "impressive but frustrating" to genuinely excellent.

## Strukturelle temaer
### 1. No shared Customer→Site→Host identity spine (the root cause)
_Berorer: Compliance, Fleet, Provisioning, Security, SSH, ScopingModel_

The single highest-leverage problem. A host has four uncrossable identities: free-text group (SshHost.swift:10, filtered group==slug at ContentView:895 / ComplianceListColumn:33), an IP appended into Site.hostIds (DiscoveryPanel:944, CustomerEditSheet:597) that every reader matches as a record id (ProvisioningView:350,374), an IP-keyed findings/risk store (HostRisk.hostIp), and a record-id-keyed compliance store (complianceHistory keyed by hostId, AppState:807). Nothing reconciles them, and there is no UI to attach a real host record to a site. This directly produces: Compliance's 'No compliance-capable hosts' for customers with FortiGates; Fleet being blind to compliance scores and mislabeling compliance-only customers 'never scanned'; Provisioning diff/deploy permanently disabled for any auto-discovered host (IP can never == record id); and 'Add as SSH host' hardcoding group:'Discovered' (DiscoveryPanel:929), severing a discovered host from its customer at the one moment the app knows the customer. Every per-tab silo finding is a leaf of this root.

### 2. Two customer-scope variables + no selection reconciliation on scope change
_Berorer: ScopingModel, InfoArchitecture, Provisioning, Security, Fleet, VPN, Recon_

globalCustomerSlug (AppState:927, a plain stored var with NO didSet/onChange anywhere) filters four tabs' lists (SSH/Compliance/Security/Fleet-Activity) but never reconciles the detail selection, so switching customer leaves stale detail panes live — Security can run an active scan against customer X while the list says Y (SecurityView:14-17 resolves against the UNFILTERED engagements; FleetView never reacts at all). Meanwhile Provisioning maintains a SECOND scope var, selectedCustomerSlug (AppState:932), never synced in either direction, forcing operators to re-pick the customer they already chose and meaning a newly-created customer doesn't scope anywhere else. A third surface (Recon's UniFi-adopt, NetworkScanSheet:1263) reads the wrong one. VPN/Tailscale/Recon ignore the picker entirely; on VPN the picker is purely decorative (filteredVpnProfiles, ContentView:500-517, only filters search text; VpnProfile has no customer field).

### 3. Dead detail panes — prime real estate is empty on entry in 6 of 8 sections
_Berorer: EmptyStates, SSH, VPN, Tailscale, Compliance, Security, Provisioning_

The detail column is the app's largest panel (min 480 / ideal 720pt, ContentView:56-58). On entry to SSH (404-412), VPN (416-425), Tailscale (435-444), Compliance (449-458), Security (SecurityView:34-43), and Provisioning (two stacked placeholders, ProvisioningView:118-138) it shows a centered glyph + 'Select X to begin' with no action. In every case the data to fill it is already loaded: tailscaleStatus carries the full tailnet snapshot; vpnConnectionStates + reachableVpnNetworks() are live; complianceHistory holds per-host scores; and Security's own loadFindingsSummary/loadHostRisks/loadActivityTimeline are wired up and consumed by Fleet but never called by Security. Fleet proves the dashboard pattern works; the other six just don't use it. The empty panes are also hand-rolled VStacks, weaker than the app's own good ContentUnavailableView CTAs used in the sidebar lists.

### 4. Recon + Security are one engagement workflow split across two tabs with lossy hand-offs
_Berorer: Recon, Security, InfoArchitecture, Provisioning_

Both call the same engine RPC (discovery_active_scan); the app spends a whole explanatory card (whereIsActiveScanCard, ReconView:266-284) telling the operator the two front doors are equivalent. The DNS-audit and traffic-capture sheets physically live in Views/Security/ but are only presented from Recon (ReconView:305,312), forcing a tab-hop mid-task. Worse, the hand-offs are asymmetric and lossy: Security→Recon carries context via pendingReconEngagementId (with a code comment warning that losing it files against the wrong scope), but Recon→Security ('Open in Security', NetworkScanSheet:210) only flips the section and drops the engagement (Recon's selectedEngagementId is view-local; Security reads the global appState.selectedEngagementId). Recon's 'Open Provisioning…' (NetworkScanSheet:142) discards the host it was invoked on entirely, and 3 of 5 Recon tools silently ignore the engagement picker the UI promises scopes them.

### 5. Half-built signals and dropped inputs that read as broken or are silently wrong
_Berorer: SSH, Provisioning, Compliance, Tailscale, Fleet_

Several wired-up affordances do nothing or the wrong thing. hostHealth (AppState:18) is read at ContentView:1008 but written nowhere — every SSH host shows a permanently gray dot, backed by a Settings interval picker that drives no poller. Provisioning extras (PPPoE creds etc.) are passed to render but NOT to diff or deploy (DiffPreviewSheet constructed without them, ProvisioningView:99-105) so you deploy a different config than you rendered/reviewed. Provisioning's rendered output is view-local @State that never resets on site switch, so Copy/Save/Deploy can act on Site A's config while the header says Site B. customer.defaultTemplate is read in four places but unsettable in the GUI. Compliance's entire historical-run UI (historyStrip:747, clickable trend bars) is dead code — the body renders trendChartSection with no selection handler. EditHostSheet downgrades the customer Group picker to a bare TextField (line 37), a data-integrity hole. TailscaleDetailView lacks .id(peerId) so ping latency bleeds between peers.

### 6. One-shot tools with no baseline / diff / scheduling, and an under-adopted design system
_Berorer: Recon, Compliance, Provisioning, Security, Fleet, VisualSystem_

Across Recon, Compliance, Provisioning, and Security's recon-style tools, scans are fire-and-forget: re-running clobbers the prior result with no 'what changed since last sweep' (NetworkScanSheet:313 wipes result; DNS-audit/health keep findings only in @State and discard on close). Compliance scheduling is a single global FortiGate-only Settings toggle with no per-customer/per-host cadence; Provisioning has no drift baseline. Fleet has no background refresh or sidebar badge despite the NotificationManager + poller plumbing existing. Separately, the shared component set (SeverityBadge, VpnStatusDot, DispositionLabel) is genuinely good but under-adopted — Fleet and Compliance re-hardcode severity colors at drifting opacity (0.15 vs 0.18), three VPN status-color palettes disagree on what 'connecting' looks like (yellow vs orange), status dots are hand-rolled in five places, and every tab invents its own section header in a different place at a different size.

## Prioritert roadmap (impact / effort)

**1. [HIGH/M] Build a Swift-side HostIndex that reconciles record-id ↔ ip ↔ group ↔ siteId once per refresh**  (ScopingModel / AppState)  
This is the root-cause fix and the highest leverage change in the app. Even before any engine change, a single resolver on AppState (built from sshHosts: hostname==ip, group) that all four scope-resolvers consult would let Fleet join compliance+security by host, let Compliance include site-attached hosts, and let Provisioning resolve discovered hosts. It directly dissolves the owner's headline symptom and unblocks findings #2-#6 below. Without it you keep patching leaves. Medium effort because it's additive (a derived index) rather than an engine schema migration.

**2. [HIGH/S] Thread Provisioning extras into diff + deploy (DiffPreviewSheet init param)**  (Provisioning / DiffPreviewSheet:433-457, ProvisioningView:99-105)  
A correctness landmine on a tool that pushes firewall CLI over SSH: the operator renders a config WITH their PPPoE/S2S creds, reviews it, then the diff and deploy run WITHOUT them (both default extras to [:]). You review one config and push another. Pure plumbing — add an extras param and forward it — but the payoff is 'what I reviewed is what I deploy.' Ship before anyone deploys a real branch-office config.

**3. [HIGH/S] Reset Provisioning render state on site/customer change (.onChange or .id keyed on selection)**  (Provisioning / ProvisioningView:26-30,49-78)  
selectedTemplateId/extras/rendered are view-local @State on a persistent view with no reset, so after rendering Site A and clicking Site B the header shows B while Copy/Save/Deploy act on A's stale output. Worst case deploys the previous site's config to the wrong site. One onChange handler (or .id("slug-site")) closes a wrong-target risk on the deploy path.

**4. [HIGH/S] Make Compliance route any compliance-capable host (drop the FortiGate-only detail gate)**  (Compliance / ContentView:446-447)  
The list deliberately includes Linux hosts (and tells operators to add them), but the detail gate is deviceType == .fortigate, so selecting a Linux row dead-ends on 'Select a FortiGate host' and the entire Linux CIS scan path is unreachable from the UI. Change the gate to deviceType.complianceDispatch != .notApplicable (the same allowlist the list uses). One-line fix that turns a whole feature back on.

**5. [HIGH/M] Fold compliance into Fleet rows + fix 'never scanned' to span both stores**  (Fleet / FleetView:248-261,309-317,420-423)  
Fleet — the morning-coffee 'who's red' board — pulls only the Security findings store, so a FortiGate that just failed CIS at score 40 shows a clean/low card, and a compliance-only customer reads 'never scanned' in alarm-orange forever. The single most actionable red signal the toolkit produces never reaches the dashboard meant to surface it. Depends on rank-1's join key; add a parallel compliance pull and drive topRiskBand off max(security, compliance).

**6. [HIGH/S] Pass the engagement's customerSlug when promoting a discovered host (not group:'Discovered')**  (Security / DiscoveryPanel:929, plus IP→record-id on link)  
The one moment the app has full context (engagement, customer, IP, device type) to wire a host into the customer model correctly, it throws the customer away with a hardcoded literal — so the host NEVER appears in Compliance or the SSH list scoped to that customer. Also guard against the unconditional addHost that duplicates a 'Discovered' host on every re-scan. Small change, large correctness payoff, and it stops accumulating duplicate hosts on weekly sweeps.

**7. [HIGH/M] Default the six dead detail panes to section dashboards (start with Security — loaders already exist)**  (EmptyStates / SecurityView:34-43, ContentView:404-457)  
Six of eight sections open dead in the 720pt pane. Security is the cheapest, highest-value start: loadFindingsSummary/loadHostRisks/loadActivityTimeline are already written and proven by Fleet, just never called here. Tailscale and VPN are nearly free too (tailscaleStatus / vpnConnectionStates already in memory). Convert at minimum to ContentUnavailableView with the section's primary action so the operator gets a button, not a sentence.

**8. [HIGH/L] Collapse Recon into Security as one engagement workflow; route tools inline**  (InfoArchitecture / ReconView, SecurityView, DiscoveryPanel)  
Recon and Security are scope→scan→triage→report for the same engagement on the same RPC, split across two tabs with a whole card apologizing for the duplication and DNS/capture sheets that live in Security but only open from Recon. Merging deletes the lossy cross-tab hand-offs (the engagement-drop and host-drop bugs), removes whereIsActiveScanCard, and drops the sidebar to 7. Larger effort and lower urgency than the data-spine fixes, but it's the right structural end-state and resolves a cluster of medium findings at once.

**9. [HIGH/M] Give globalCustomerSlug a didSet that reconciles per-section selection; merge in Provisioning's selectedCustomerSlug**  (ScopingModel / AppState:927,932)  
The picker is a plain var with no observer, so switching customer strands stale detail panes (Security scans X while list says Y) and Provisioning's parallel scope var forces re-picking the customer every crossing. A didSet that clears selectedHostId/selectedEngagementId/selectedSiteId when they leave scope, plus collapsing selectedCustomerSlug into the global, removes a confusing class of stale-pane and re-pick friction in one place.

**10. [HIGH/M] Add a host-attach control to the site editor + resolve IP→record-id in both Site.hostIds writers**  (Provisioning / CustomerEditSheet siteEditor, DiscoveryPanel:944, CustomerEditSheet:597)  
Both writers store scan IPs while every reader matches record ids, so Provisioning diff/deploy is permanently disabled for auto-detected/discovered hosts and there's no UI to attach a real host to a site at all. Resolve the IP to a record id (or create the record) before appending, and add a multi-select host picker to siteEditor. This is the concrete Site→Host edge of the rank-1 spine and re-enables deploy for the discovery-driven workflow the app clearly wants operators to use.

**11. [MEDIUM/M] Wire the SSH host-health dot to a real poller (or remove dot + Settings picker)**  (SSH / AppState:18, ContentView:1007-1010, GeneralSettingsView:132)  
Every host shows a permanently gray status dot (hostHealth is read once, written nowhere) backed by a Settings interval that drives nothing. A column of meaningless gray dots plus a knob that does nothing reads as 'all hosts unknown/down' — worse than no signal. Either back it with a cheap TCP-connect poll on the configured interval, or delete both until it's real. Half-built health signaling actively misleads the daily operator.

**12. [MEDIUM/M] Surface compliance baseline + firmware in the header, history opener, and de-dupe score/severity tokens**  (Compliance + VisualSystem / ComplianceHostView:159,747, SeverityBadge)  
Bundles the under-adopted design-system cleanups with two Compliance gaps: the header never shows which baseline/firmware a score was measured against (matters because CIS applicability is firmware-dependent), and the entire historical-run UI is dead code (historyStrip/clickable trend bars referenced nowhere; trendChartSection has no selection handler) so operators can only ever see the latest run. Route Fleet/Compliance severity pills through SeverityBadge.color(for:) and extract one scoreColor/grade helper to stop the 0.15-vs-0.18 and 90/70-in-six-places drift while you're in here.
