# HostIndex Design — unified Customer->Site->Host resolver

Generated 2026-06-11 from a grounded trace of all four host-identity schemes.
Fixes the #1 structural defect from UX-AUDIT-2026-06-11.md.

## Resolver design

WHAT IT IS: `HostIndex` is a value type (immutable struct) stored on AppState as a single derived property `var hostIndex: HostIndex` (AppState is `@Observable`, AppState.swift:7-8, so reassigning it triggers SwiftUI updates). It is NOT a new persisted model and changes no wire shape. It is rebuilt from the two stores that already exist in memory: `sshHosts: [SshHostSummary]` (AppState.swift:16) and `customers: [Customer]` (AppState.swift:919).

WHAT IT RECONCILES: the single object that simultaneously knows all four keys is `SshHostSummary` — it carries the record id (`id`, SshHost.swift:5; the axis-2/axis-4 key), the IP (`hostname`, SshHost.swift:7; the axis-3 key, since both discovery writers store hostname=host.ip — DiscoveryPanel.swift:926, NetworkScanSheet.swift:1322 — and NetworkScanSheet.swift:1332 already joins on exactly this), and the customer string (`group`, SshHost.swift:10; the axis-1 key). The index folds in the only structural Customer->Site->host edge, `Site.hostIds` (ProvisioningModels.swift:24), which on disk holds IPs (DiscoveryPanel.swift:944, CustomerEditSheet.swift:597) but is intended as record ids. The index therefore tolerates a `hostIds` entry that is EITHER an id OR an IP.

INTERNAL SHAPE (all built once per refresh):
 - `byId: [String: SshHostSummary]` — record id -> host (the canonical axis).
 - `byIp: [String: SshHostSummary]` — host.hostname -> host (bridges axis-3 IPs and IP-valued hostIds).
 - For each `Customer` x `Site`, walk `site.hostIds`; resolve each token through `byId[token] ?? byIp[token]` to a real host; record `recordId -> (slug, siteId)` and accumulate `slug -> Set<recordId>` and `slug -> Set<ip>`.
 - `slugForHost: [String /*recordId*/ : String /*slug*/]` resolved by precedence: (1) exact `group == slug` for a known customer slug; else (2) the host is reachable through some `Site.hostIds` (by id or ip) -> that customer's slug; else (3) nil (genuinely ungrouped).

KEY LOOKUPS THE INDEX EXPOSES (pure, no I/O):
 - `host(forToken:)` — id-or-ip -> SshHostSummary?. This is the axis-2 fix: ProvisioningView's `$0.id == hostId` (ProvisioningView.swift:373,384,397) becomes `hostIndex.host(forToken: hostId)`, so an IP-valued hostIds entry now resolves, and the returned `host.id` is always a real record id for downstream daemon calls (DiffPreviewSheet.swift:440,460 stay correct).
 - `customerSlug(forHost:)` — SshHostSummary -> slug?. This is the axis-1 fix: ContentView.swift:926 and ComplianceListColumn.swift:33 change `$0.group == global` to `hostIndex.customerSlug(forHost: $0) == global`, so a FortiGate with group:"Discovered" (DiscoveryPanel.swift:929) but linked by IP in Site.hostIds (DiscoveryPanel.swift:944) now appears.
 - `hostsForCustomer(slug:)` and `recordIds(forCustomer:)` / `ips(forCustomer:)` — let Fleet (FleetView.swift:405-430) fold the record-id-keyed compliance store (complianceHistory, AppState.swift:807) into a card that today only reads the IP-keyed security store, and reconcile a HostRisk.hostIp (SecurityModels.swift:546) back to a record id via `byIp`.

HOW IT IS BUILT AND REFRESHED: add `func rebuildHostIndex()` on AppState that sets `hostIndex = HostIndex(hosts: sshHosts, customers: customers)`. Call it at the tail of `refreshHosts()` (AppState+SSH.swift:8-14) and `refreshCustomers()` (AppState+Provisioning.swift:6-12) — every existing write path already funnels through these two (addHost/updateHost/togglePin -> refreshHosts at AppState+SSH.swift:46,80,89; saveCustomer -> refreshCustomers at AppState+Provisioning.swift:25), and `refreshAll()` (AppState.swift:457,461) calls both at launch, so the index is always current with zero new call sites in the views. Cost is O(hosts + sites) over a handful of in-memory arrays; negligible.

## First PR (smallest visible win)

Step 1 + Step 2 shipped together: introduce HostIndex (built at the tail of refreshHosts/refreshCustomers) and flip the single Compliance filter at ComplianceListColumn.swift:33 from `$0.group == global` to `appState.hostIndex.customerSlug(forHost: $0) == global`. This is the smallest change that produces a visible win — Compliance stops showing "No compliance-capable hosts" (the empty state at ComplianceListColumn.swift:55,65) for any customer whose FortiGate exists in Provisioning but carries group:"Discovered" or is linked only by IP in Site.hostIds. It is safe because: (a) HostIndex is a pure additive value type that nothing else reads yet, so Step 1 alone is a no-op; (b) the resolver's first precedence arm is exact `group == slug`, so every host that matches today still matches (no regression to correctly-grouped fleets); (c) it touches one filter expression and no persisted shape, daemon call, or model field — fully reversible. The same one-line resolver swap is then repeated in Steps 3-4 for the SSH sidebar and Provisioning, but Compliance is the highest-visibility, lowest-blast-radius first cut because the empty-state is the headline symptom of the defect.

## Risks
- Site.hostIds token ambiguity: an entry could in theory collide between byId and byIp namespaces. byId is checked first (record ids are daemon UUIDs, AppState+SSH.swift:45), and IPs never look like those UUIDs, so collision is effectively impossible — but the precedence (`byId[token] ?? byIp[token]`) must stay id-first to be safe.
- group==slug precedence vs stale links: if a host's group is a real-but-WRONG slug (operator typo of another customer's slug), precedence arm (a) wins over the correct Site.hostIds link, keeping the host mis-attributed. The IP/site link only fills gaps; it does not override an explicit (even if wrong) group. Acceptable for v1; Step 7 (slug Picker) prevents new typos.
- Duplicate IPs across customers: if the same IP appears in two customers' Site.hostIds (e.g. overlapping RFC1918 ranges, or a re-used host), byIp maps one IP to one host and slugForRecordId takes the last writer in customer iteration order — last-write-wins is nondeterministic across refreshes. Low likelihood in this fleet but should be noted; a future version could detect and surface the conflict.
- Index staleness if a new mutation path bypasses refreshHosts/refreshCustomers: the design relies on every write funneling through those two refreshers (verified true today). Any future code that mutates sshHosts/customers directly must also call rebuildHostIndex(), or readers see a stale index. Mitigation: keep the `let` arrays private-set and rebuild centrally.
- Fleet compliance fold-in (Step 5) reads complianceHistory which is populated lazily/on-demand for some surfaces; a customer card may show a stale or missing compliance score until an auto-scan (kickComplianceAutoScanIfDue) or view-entry hydrate runs. Step 5 should treat absent compliance as 'unknown', not 'clean', to avoid re-introducing the false-clean problem it set out to fix.
- Behavioral change visibility: Steps 2-4 make previously-hidden hosts appear. Operators accustomed to the empty Compliance/sidebar may see hosts they consider 'not theirs' if Site.hostIds links are sloppy. Worth a brief note in release notes; the data was always linked, just unreadable.
